# Abilities Plugin: CLI Credential Sharing Requirements

**Related issue**: abilityai/trinity#259
**Target repo**: [abilityai/abilities](https://github.com/abilityai/abilities) (trinity-onboard plugin)

## Summary

Update the `trinity-onboard` plugin to detect and use credentials from the Trinity CLI (`~/.trinity/config.json`) instead of requiring manual configuration. Deploy should delegate to `trinity deploy .` instead of reimplementing tar+upload via MCP.

## Requirements

### 1. Credential resolution from CLI profile

**File**: `trinity-onboard/SKILL.md` (onboard skill)

When the plugin needs Trinity credentials (instance URL, API key), resolve in this order:

1. `TRINITY_API_KEY` environment variable (existing behavior)
2. `~/.trinity/config.json` active profile (`token` + `instance_url`)
3. Prompt user interactively (existing fallback)

**Implementation**:
- Read `~/.trinity/config.json`, parse `current_profile`, get matching profile data
- Use `instance_url` as the Trinity instance
- Use `token` as the Bearer auth token
- If `mcp_api_key` is present in profile, use it for MCP server configuration

### 2. Deploy via CLI command

**File**: `trinity-onboard/SKILL.md` (onboard skill, deploy step)

The deploy step should call `trinity deploy .` (the CLI command) instead of:
- Manually tar+base64 encoding the directory
- Calling MCP `deploy_local_agent` tool directly

**Benefits**:
- Single deploy implementation (CLI owns it)
- Automatic `.trinity-remote.yaml` tracking
- Automatic versioning and previous-version stopping
- Consistent behavior between CLI and plugin

**Fallback**: If `trinity` CLI is not installed, fall back to MCP-based deploy (existing behavior).

### 3. Shared tracking file

**File**: `.trinity-remote.yaml`

Both the CLI and the plugin should read/write the same tracking file:

```yaml
# Auto-generated by trinity deploy — do not edit
instance: https://trinity.example.com
agent: my-agent
profile: production
deployed_at: 2026-04-03T15:45:00Z
```

- On deploy: write `.trinity-remote.yaml` with instance, agent name, profile, timestamp
- On redeploy: read `.trinity-remote.yaml` to determine target agent name
- If instance in tracking file differs from current context, warn user

### 4. MCP key from profile

If `~/.trinity/config.json` has `mcp_api_key` in the active profile (provisioned by `trinity init`):
- Use it when configuring MCP server connection in `.mcp.json`
- Skip prompting user for MCP API key
- Still allow explicit `TRINITY_API_KEY` env var to override

## Acceptance Criteria

- [ ] `trinity-onboard:onboard` detects `~/.trinity/config.json` and skips auth prompts when profile exists
- [ ] Deploy step calls `trinity deploy .` when CLI is installed
- [ ] Falls back to MCP deploy when CLI is not available
- [ ] `.trinity-remote.yaml` is read/written consistently with CLI behavior
- [ ] MCP key from profile is used for `.mcp.json` configuration
- [ ] `TRINITY_API_KEY` env var still overrides profile credentials

## Dependencies

- Trinity CLI `trinity deploy .` command (abilityai/trinity#259) must be merged first
- `~/.trinity/config.json` profile format with `mcp_api_key` field must be stable
